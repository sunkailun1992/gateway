# 网关安全编码规范

本文件只维护网关安全规则。AI 新增或修改路由、跨域、请求头透传、日志、Knife4j、监控入口、代理转发时，必须先阅读本文件。

## 接口鉴权和网关边界

- 网关不实现业务鉴权，但必须原样透传 `Authorization`、租户、链路追踪等必要请求头，让后端服务完成认证授权。
- 禁止新增任何 token、用户、权限或业务响应解析逻辑；业务安全判断必须在后端服务中完成。
- 新增路由时必须检查是否会绕过后端服务的安全过滤器，禁止通过改写路径访问未授权的内部端点。
- 退出登录、刷新 token、token 清理和用户禁用后的失效逻辑属于认证服务职责；网关只能透传对应请求，不能缓存、伪造、复用或自行清理业务 token。
- 退出清理 token 由认证服务落地，网关不得通过缓存、重试、降级或请求头重写让旧 token 继续访问接口。
- 网关新增公开路由时必须明确公开原因、目标服务、允许方法、来源限制和限流策略。
- 登录、刷新 token、退出登录、上传、导出、批量操作等高风险路由必须支持限流、请求体大小限制和基础审计，防止资源消耗和业务流程滥用。
- 如果网关允许 Cookie 或浏览器自动携带凭证跨域访问，必须评估 CSRF 风险，并通过 SameSite、CSRF Token、Origin/Referer 校验或 Bearer Token 模式降低风险。

## 路由暴露

- 路由和 RewritePath 规则必须最小化匹配范围，禁止把管理接口、内部接口、Actuator、Nacos、Seata、XXL-JOB、Kibana 等运维入口意外暴露到公网路径。
- Knife4j、Actuator、Admin、Zipkin 等调试或监控入口必须明确运行环境和访问控制，不得默认在生产公网开放。
- 新增微服务路由时，必须确认目标服务自身已启用认证授权，不能用网关路由替代后端权限。
- 路由改写不得让调用方通过替换路径、服务名、版本号或前缀访问其他租户、其他服务、内部接口或未授权资源，防止水平越权入口绕过。
- 文件下载、静态资源、预览和代理类路由必须限制路径前缀和目标服务，不能把任意路径映射到后端文件系统或内部网络。
- 路由默认失败关闭；目标服务、鉴权头、租户头、签名或限流组件异常时不得降级为放行。
- Nacos 路由配置必须可审计、可回滚，新增、删除或放宽路由时必须记录原因和验证结果。

## CORS 和请求头

- CORS 配置必须限制来源、方法和请求头；不得为了调试长期保留 `*` 放开所有来源和凭证组合。
- 只透传认证、租户、链路追踪和业务必需请求头，不新增无来源校验的用户、角色、权限头。
- 对外响应头不得暴露内部服务地址、容器名、数据库地址或异常堆栈。
- 数据脱敏同样适用于网关响应、异常响应和日志字段，不能把后端敏感信息原样透出到浏览器或日志。
- 不允许把前端传入的 `userId`、`role`、`permission`、`tenantId` 等身份类头作为可信鉴权事实生成或覆盖后端认证上下文。
- 允许跨域携带凭证时必须配置明确来源白名单，不能与任意来源通配同时出现。
- 响应头必须按需设置安全头，例如 `X-Content-Type-Options`、`Referrer-Policy`、`Content-Security-Policy`、`X-Frame-Options` 或等价配置，避免 MIME 嗅探、点击劫持和脚本注入风险。
- 不得透传或生成包含密钥、内部服务名、调试开关、灰度规则、数据库地址的响应头。
- 加密、密钥、批量赋值和字段级授权属于后端服务职责；网关不得通过请求头重写、路径改写、参数注入或降级转发绕过后端的字段级授权、批量赋值防护、签名校验和加密密钥保护。
- 敏感字段明文查看必须由后端服务提供专用接口、专用权限、字段级授权、审计日志和二次确认；网关不得新增绕过后端授权的明文查看路由、解密逻辑、请求头开关或参数开关。
- 网关日志、错误响应、限流记录、链路追踪和访问日志不得记录解密后的手机号、邮箱、身份证号、银行卡号、详细地址、token、验证码或消息正文。

## 日志和代理安全

- 限流、跨域、日志、错误处理不得输出 token、cookie、密钥、完整手机号、完整邮箱、数据库连接串和内部服务地址。
- 网关日志不得记录完整请求体；涉及登录、上传、消息内容、个人资料等接口时只记录必要元数据。
- 文件下载、静态资源、代理转发类需求必须防止路径遍历和开放代理；用户输入不得直接作为目标 URL、目标路径或上游服务名。
- 文件遍历防护必须覆盖网关路径改写、静态资源代理、下载预览代理和上游路径拼接，不能让 `../`、绝对路径或编码绕过进入后端。
- 禁止新增可由外部参数控制的任意转发地址，防止 SSRF 和开放重定向。
- 网关不得把未清洗的请求参数、路径、文件名或错误信息拼接进 HTML 响应、JavaScript、重定向地址或响应头，防止 XSS 跨站脚本和响应头注入。
- 文件上传校验在网关侧至少要限制大小、方法和目标服务；文件类型、扩展名、MIME、文件头和存储权限由后端服务继续校验，网关不能绕过。
- 涉及查询参数透传时，网关不得新增把用户输入拼接成 SQL、排序字段或数据库片段的过滤器；SQL 注入防护由后端参数绑定和字段白名单完成。
- 错误页、降级响应和网关异常响应必须使用固定模板，不回显未转义的用户输入。
- SSRF 防护必须覆盖任何可由外部参数影响的代理、预览、回调、跳转和重定向场景，禁止访问内网地址、本机地址、云元数据地址和基础设施管理入口。
- 代理请求必须设置连接超时、读取超时、最大响应体、重定向限制和失败降级，不得提供无超时的任意转发能力。

## 安全配置和供应链

- Spring Cloud Gateway、Knife4j、Actuator、CORS、限流、错误响应、日志和 Nacos 配置必须按最小暴露原则维护，生产环境不得默认开放调试入口。
- 新增依赖、插件、镜像、脚本和 Gradle/Maven 仓库时必须说明用途，优先使用可信仓库和稳定版本，避免引入未维护、漏洞严重或来源不明的组件。
- 依赖升级必须关注安全公告和破坏性变更，不得为了追最新版本跨大版本引入不兼容风险。
- CI/CD、Nacos 配置、路由配置和启动参数不得明文写入密钥、token、数据库密码或内部访问凭证。

## 完整性、异常和安全日志

- 网关异常处理必须失败关闭，路由、限流、CORS、签名、鉴权头校验异常不得降级为允许访问。
- 网关不得反序列化不可信对象、脚本、表达式、模板或规则；动态路由和过滤器配置必须来自可信配置源。
- 权限拒绝、路径穿越、SSRF 命中、开放重定向拦截、CORS 拒绝、限流触发、异常上传、SQL 注入和 XSS payload 命中必须记录安全审计日志。
- 安全日志必须包含时间、来源 IP、请求路径、目标服务、结果、原因和 traceId，但不得包含 token、cookie、请求体敏感内容或内部连接串。
- 高风险安全事件必须具备告警或可检索标识，不能只写普通业务日志后无人关注。

## 验证要求

- 修改网关安全相关配置后，必须验证未登录、无权限、跨服务路径改写和 CORS 预检场景。
- 新增路由后必须验证目标服务未登录访问被拒绝，不能只验证网关转发成功。
- 修改日志、错误处理或过滤器后，必须确认响应和日志不包含 token、cookie、内部 IP、内部域名、数据库连接串和堆栈。
- 文件、上传、下载、预览、代理和重定向相关路由必须验证路径穿越、非法扩展名、超限请求体、开放代理和恶意跳转地址。
- 安全测试必须覆盖 XSS payload、SQL 注入 payload、伪造身份请求头、跨租户路径和退出登录后旧 token 访问。
- 安全测试还必须覆盖 CSRF、SSRF、限流、资源消耗、失败关闭、安全响应头、CORS 凭证组合、依赖风险和安全日志审计。
