# 网关安全编码规范

本文件只维护网关安全规则。AI 新增或修改路由、跨域、请求头透传、日志、Knife4j、监控入口、代理转发时，必须先阅读本文件。

## 网关边界

- 网关不实现业务鉴权，但必须原样透传 `Authorization`、租户、链路追踪等必要请求头，让后端服务完成认证授权。
- 禁止新增任何 token、用户、权限或业务响应解析逻辑；业务安全判断必须在后端服务中完成。
- 新增路由时必须检查是否会绕过后端服务的安全过滤器，禁止通过改写路径访问未授权的内部端点。

## 路由暴露

- 路由和 RewritePath 规则必须最小化匹配范围，禁止把管理接口、内部接口、Actuator、Nacos、Seata、XXL-JOB、Kibana 等运维入口意外暴露到公网路径。
- Knife4j、Actuator、Admin、Zipkin 等调试或监控入口必须明确运行环境和访问控制，不得默认在生产公网开放。
- 新增微服务路由时，必须确认目标服务自身已启用认证授权，不能用网关路由替代后端权限。

## CORS 和请求头

- CORS 配置必须限制来源、方法和请求头；不得为了调试长期保留 `*` 放开所有来源和凭证组合。
- 只透传认证、租户、链路追踪和业务必需请求头，不新增无来源校验的用户、角色、权限头。
- 对外响应头不得暴露内部服务地址、容器名、数据库地址或异常堆栈。

## 日志和代理安全

- 限流、跨域、日志、错误处理不得输出 token、cookie、密钥、完整手机号、完整邮箱、数据库连接串和内部服务地址。
- 网关日志不得记录完整请求体；涉及登录、上传、消息内容、个人资料等接口时只记录必要元数据。
- 文件下载、静态资源、代理转发类需求必须防止路径遍历和开放代理；用户输入不得直接作为目标 URL、目标路径或上游服务名。
- 禁止新增可由外部参数控制的任意转发地址，防止 SSRF 和开放重定向。

## 验证要求

- 修改网关安全相关配置后，必须验证未登录、无权限、跨服务路径改写和 CORS 预检场景。
- 新增路由后必须验证目标服务未登录访问被拒绝，不能只验证网关转发成功。
- 修改日志、错误处理或过滤器后，必须确认响应和日志不包含 token、cookie、内部 IP、内部域名、数据库连接串和堆栈。
